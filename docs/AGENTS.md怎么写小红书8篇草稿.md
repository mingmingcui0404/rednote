# AGENTS.md 怎么写：小红书 8 篇草稿

## 1. AGENTS.md 是什么？给 Codex 一份项目说明书

很多人第一次用 Codex，会反复遇到一个问题：

每次都要重新解释项目结构、运行命令、代码风格和注意事项。

后来我发现，这些重复说明不应该每次都写在聊天里，而应该沉淀到 `AGENTS.md`。

你可以把 `AGENTS.md` 理解成：写给 Codex 的项目说明书。

它不负责替你写需求，而是告诉 Codex：

这个项目是做什么的。
哪些文件最重要。
常用命令是什么。
修改代码时要遵守什么规则。
哪些动作必须先问你。

比如一个前端项目，可以在里面写：

修改前先读 `index.html`、`styles.css`、`app.js`。
改完后运行 `node --check app.js`。
不要重构无关代码。
涉及发布、删除、推送必须先确认。

这样下次你让 Codex 改项目，它就不用从零开始猜。

真正好用的 AI 协作，不是每次把提示词写得很长，而是把长期规则放到它能自动读取的位置。

`AGENTS.md` 就是这个位置。

## 2. AGENTS.md 应该写什么？新手先放这 7 类信息

如果你不知道 `AGENTS.md` 从哪里开始写，可以先放 7 类信息。

第一类：项目概览。

用几句话说明这个项目是做什么的，面向谁，核心功能是什么。

第二类：文件结构。

告诉 Codex 哪些文件是入口文件，哪些目录不要乱动。

第三类：常用命令。

比如启动、测试、构建、语法检查命令。

第四类：编辑规则。

比如优先小范围修改，不做无关重构，不删除用户已有内容。

第五类：代码风格。

比如命名方式、组件写法、CSS 习惯、是否使用 TypeScript。

第六类：验证方式。

改完之后应该跑什么检查，或者怎么人工验证。

第七类：高风险动作。

发布、删除、推送、提交表单、修改线上配置，都应该先确认。

新手不要一开始写得特别长。

`AGENTS.md` 最重要的不是完整，而是能减少 Codex 的猜测。

先写一版能用的，再随着项目慢慢补。

## 3. 一个好用的 AGENTS.md 模板，直接照着填

如果你想快速开始，可以直接用这个结构：

```md
# AGENTS.md

## 项目概览

这是一个用于 [项目用途] 的项目。请先理解现有结构，再进行修改。

## 重要文件

- `[文件路径]`：负责 [作用]
- `[文件路径]`：负责 [作用]

## 常用命令

- 安装依赖：`[命令]`
- 本地运行：`[命令]`
- 语法检查：`[命令]`
- 测试：`[命令]`
- 构建：`[命令]`

## 编辑规则

- 优先沿用现有结构和命名方式。
- 不做与当前任务无关的重构。
- 不删除用户已有内容，除非任务明确要求。
- 修改前先阅读相关文件。

## 验证要求

完成修改后，运行最轻量的可用检查。
如果无法验证，请说明原因和残余风险。

## 高风险动作

以下动作必须先获得用户确认：

- 删除文件
- 发布内容
- 推送代码
- 提交表单
- 修改线上配置
```

这份模板不复杂，但够用。

它解决的是最基础的问题：让 Codex 知道项目边界。

别急着把所有规则都写进去。

先把“它最容易猜错的地方”写清楚，就已经能稳定很多。

## 4. AGENTS.md 不该写什么？这 5 类内容别放进去

`AGENTS.md` 很有用，但不是所有东西都适合写进去。

第一类：密钥、token、cookie。

这些永远不要放进项目说明里。Codex 不需要知道你的账号秘密。

第二类：一次性临时任务。

比如“今天帮我发一篇小红书”，这种适合写在当前对话，不适合沉淀成长期规则。

第三类：太长的业务资料。

如果资料很长，最好放到单独文档里，需要时再让 Codex 阅读。

第四类：空泛口号。

比如“代码要优雅”“体验要高级”。这类话听起来对，但不够可执行。

更好的写法是：

按钮文字不能溢出。
移动端不能横向滚动。
改完后运行指定检查命令。

第五类：互相冲突的规则。

比如一处写“必须重构”，另一处写“不要重构”，Codex 会很难判断。

`AGENTS.md` 不是许愿池。

它应该像一份清楚的工作说明：具体、稳定、可执行。

## 5. 前端项目的 AGENTS.md 怎么写？

如果你用 Codex 做前端项目，`AGENTS.md` 里一定要写清楚 4 件事。

第一，先看哪些文件。

比如：

请先阅读 `index.html`、`styles.css`、`app.js`，再判断怎么修改。

第二，设计和交互原则。

比如：

不要做空泛的 landing page。
第一屏要直接呈现核心功能。
移动端文字不能溢出。
按钮、表单、空状态要完整。

第三，修改边界。

比如：

不要无关重构。
不要改动未涉及的模块。
不要删除已有内容。

第四，验证方式。

比如：

修改 JS 后运行 `node --check app.js`。
涉及页面布局时，用浏览器检查桌面端和移动端。

我自己的经验是：前端项目最怕 Codex “发挥太多”。

所以 AGENTS.md 里要多写边界，少写审美形容词。

把“什么不能做”和“怎么验证”写清楚，页面会稳定很多。

## 6. 内容运营项目的 AGENTS.md 怎么写？

如果你的项目是小红书、公众号、短视频脚本、知识产品，`AGENTS.md` 可以这样写。

先写项目定位：

这是一个内容运营项目，重点是选题、文案、排版、发布和复盘。

再写内容风格：

语言自然。
适合手机阅读。
标题要有吸引力，但不能夸大。
正文要有具体场景和可执行建议。
标签必须相关，不堆无关热词。

然后写工作流程：

1. 明确目标用户。
2. 提炼痛点或需求。
3. 生成标题、开头、正文、标签。
4. 给出封面建议。
5. 补充发布前检查清单。
6. 如需发布，先准备草稿，最终发布前确认。

最后一定要写边界：

不自动点赞。
不自动收藏。
不自动评论。
不绕过验证码。
不确认不发布。

内容运营项目最重要的不是让 AI 写得更快，而是让它知道什么是不能碰的。

尤其是外部平台操作，一定要有确认点。

## 7. 多目录项目里，AGENTS.md 应该放在哪里？

很多人会问：`AGENTS.md` 到底放根目录，还是每个目录都放一份？

我的建议是：先根目录，再按需要加子目录。

根目录的 `AGENTS.md` 负责全局规则。

比如：

项目是做什么的。
常用命令是什么。
统一编辑原则是什么。
哪些动作必须确认。

子目录的 `AGENTS.md` 负责局部规则。

比如：

`frontend/AGENTS.md` 写前端组件规范。
`scripts/AGENTS.md` 写脚本参数和日志要求。
`docs/AGENTS.md` 写文档格式和内容风格。

规则优先级可以这样理解：

越靠近文件的规则越具体。

根目录管全局，子目录管细节。

但不要为了显得专业就到处放。

如果项目还小，一份根目录 `AGENTS.md` 就够了。

当你发现某个目录经常有特殊规则，再加子目录版本。

## 8. AGENTS.md、提示词和 Skill，到底怎么分工？

想用好 Codex，最好分清这三件东西：

提示词、`AGENTS.md`、Skill。

提示词适合一次性任务。

比如：

帮我把这篇文章改成小红书风格。
帮我修这个报错。
帮我检查这个页面。

`AGENTS.md` 适合项目长期规则。

比如：

这个项目怎么运行。
文件结构是什么。
代码怎么改。
改完怎么验证。

Skill 适合重复流程。

比如：

每次给一个方向，就生成小红书文章并准备发布。
每次做代码审查，都按固定维度检查风险。
每次打包知识产品，都输出模板、案例和清单。

可以用一句话判断：

只做一次，写提示词。
项目长期有效，写 AGENTS.md。
跨项目反复使用，沉淀成 Skill。

真正的高阶用法，不是把一句提示词写得越来越长。

而是把不同层级的信息，放到它该在的位置。
