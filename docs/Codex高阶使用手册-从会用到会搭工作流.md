# Codex 高阶使用手册：从会用到会搭工作流

这份手册适合用来做小红书系列内容、公众号长文、训练营课程大纲或知识产品目录。核心目标不是教人“多问几个提示词”，而是帮助用户把 Codex 从一次性问答工具，升级成可复用、可验证、可协作的工作流系统。

## 适合人群

- 想系统学习 Codex 的新手用户
- 内容创作者和自媒体运营者
- 独立开发者和产品经理
- 想把重复工作自动化的小团队
- 想做 AI 工作流咨询或知识产品的人

## 系列定位

一句话定位：

> 让 Codex 从“会回答问题的 AI”，变成“能接住工作的协作者”。

内容主线：

```text
Codex 入门
  ↓
AGENTS.md 项目规则
  ↓
Skill 流程沉淀
  ↓
插件与外部能力
  ↓
工作流设计
  ↓
多 Agents 协同
```

## 第一章：Codex 入门

主题：先把 Codex 当成协作者，而不是聊天框。

### 1.1 Codex 是什么

可以讲清楚：

- Codex 是面向实际工作的 AI coding agent。
- 它不只是生成代码，还能读项目、改文件、运行命令、检查结果。
- 它适合处理有明确目标、可验证结果的任务。

小红书标题示例：

- `第一次用 Codex，别把它只当聊天框`
- `Codex 真正好用的地方，不是写代码`
- `我用 Codex 后，才发现 AI 可以接住工作`

### 1.2 Codex 适合做什么

可拆成 5 类：

1. 改代码：修 bug、补功能、重构局部模块。
2. 做页面：生成静态网页、前端工具、运营工作台。
3. 写脚本：批量处理文件、生成内容、转换格式。
4. 整理资料：读 PDF、整理文档、生成结构化输出。
5. 搭流程：把重复工作沉淀成模板、Skill、自动化。

### 1.3 Codex 不适合做什么

需要明确边界：

- 不适合无确认地发布内容。
- 不适合自动刷赞、收藏、评论。
- 不适合绕过验证码、风控或平台限制。
- 不适合替代用户做最终商业判断。
- 不适合在没有上下文时盲目大改项目。

### 1.4 第一次使用建议

推荐新手从小任务开始：

- 修改一个页面文案。
- 生成一篇小红书草稿。
- 修复一个明确报错。
- 给项目补一份 `AGENTS.md`。
- 把一个重复流程整理成 checklist。

不要一开始就让 Codex：

- 重写整个系统。
- 自动上线发布。
- 处理多个不清晰目标。
- 同时改代码、写文案、发布平台。

### 1.5 好提示词结构

推荐公式：

```text
目标 + 背景 + 文件范围 + 约束 + 验证方式 + 输出格式
```

示例：

```text
请帮我优化这个小红书运营工作台。
先阅读 index.html、styles.css、app.js。
只改必要部分，不做无关重构。
新增一篇文章到内容库第一位。
修改后运行 node --check app.js。
最后告诉我改了什么和验证结果。
```

### 1.6 入门常见误区

可拆成短内容：

- 误区 1：只让 Codex 写代码，不让它先读项目。
- 误区 2：不给验证标准。
- 误区 3：一次塞太多目标。
- 误区 4：把临时偏好当成长期规则。
- 误区 5：对外部平台操作不设确认点。

### 1.7 入门案例

案例：让 Codex 修改一个静态网页并跑语法检查。

流程：

1. 让 Codex 先查看项目结构。
2. 读取相关文件。
3. 修改内容。
4. 运行语法检查。
5. 总结修改位置和验证结果。

## 第二章：AGENTS.md 怎么写

主题：让项目拥有长期记忆。

### 2.1 AGENTS.md 是什么

一句话解释：

> `AGENTS.md` 是写给 Codex 的项目说明书，用来告诉它这个项目怎么运行、怎么改、怎么验证。

它解决的问题：

- 每次不用重复解释项目结构。
- Codex 不用猜测试命令。
- 团队规范可以沉淀下来。
- 降低误改和过度重构的概率。

### 2.2 AGENTS.md 该放什么

推荐内容：

- 项目概览
- 技术栈
- 文件结构
- 常用命令
- 测试方式
- 代码风格
- 编辑边界
- 高风险动作
- 交付格式

### 2.3 AGENTS.md 不该放什么

避免写入：

- 密钥、token、cookie
- 临时任务
- 太长的业务资料
- 过时命令
- 空泛口号
- 与项目无关的个人偏好

### 2.4 推荐结构

```md
# AGENTS.md

## 项目概览

## 常用命令

## 编辑规则

## 测试要求

## 发布注意

## 高风险动作
```

### 2.5 多目录规则

可以解释：

- 根目录 `AGENTS.md` 管整个项目。
- 子目录 `AGENTS.md` 管局部规则。
- 越靠近文件的规则越具体。
- 不要在多个层级写互相冲突的规则。

### 2.6 AGENTS.md 模板方向

可做成 4 篇内容：

1. 前端项目版
2. 内容运营项目版
3. 自动化脚本项目版
4. 知识产品项目版

### 2.7 实战案例

案例：给“小红书运营工作台”写 `AGENTS.md`。

重点规则：

- 修改前先读 `index.html`、`styles.css`、`app.js`。
- 新文章加到内容库第一位。
- 修改默认内容时升级 localStorage key。
- 发布类操作必须确认。
- 不自动点赞、收藏或评论。

## 第三章：沉淀 Skill

主题：把重复流程变成可复用能力。

### 3.1 Skill 是什么

一句话解释：

> Skill 是 Codex 的任务流程说明书，用来告诉它遇到某类任务时应该怎么做。

适合写成 Skill 的任务：

- 重复出现
- 步骤相对固定
- 有明确输入和输出
- 有安全边界
- 需要工具配合

### 3.2 Skill 和 AGENTS.md 的区别

| 项目 | AGENTS.md | Skill |
|---|---|---|
| 作用 | 项目规则 | 任务流程 |
| 范围 | 某个项目或目录 | 某类可复用任务 |
| 内容 | 命令、约束、风格 | 步骤、工具、安全边界 |
| 例子 | 前端项目怎么改 | 小红书文章怎么生成并发布 |

### 3.3 什么时候该写 Skill

判断标准：

- 这个流程你已经做过 3 次以上。
- 每次都要重复解释步骤。
- 它有固定的安全边界。
- 它可能会被多个项目复用。
- 它不只是一个临时偏好。

### 3.4 Skill 的基本结构

```md
---
name: skill-name
description: 说明这个 skill 做什么，以及什么时候使用。
---

# Skill Title

## Workflow

## Writing Rules

## Tools

## Safety

## Verification
```

### 3.5 好 Skill 的标准

好 Skill 应该：

- 触发条件清楚
- 步骤可执行
- 不写废话
- 不塞太多背景
- 有安全边界
- 知道什么时候停下来问用户

差 Skill 常见问题：

- 描述太泛
- 步骤太长
- 没有输出格式
- 没有验证方式
- 没有说明高风险动作

### 3.6 Skill 的资源设计

可拆成：

- `scripts/`：稳定脚本
- `references/`：长参考资料
- `assets/`：模板、图片、字体、示例文件

判断方法：

- 每次都要重写代码 → 放脚本。
- 资料很长但不一定每次读 → 放 references。
- 需要复制使用的模板 → 放 assets。

### 3.7 实战案例：小红书发布 Skill

流程：

1. 用户给方向。
2. Codex 生成小红书文章。
3. 更新发布工作台。
4. 准备封面图。
5. 用 Chrome 打开已登录小红书。
6. 填写草稿。
7. 最终发布前确认。

安全边界：

- 不读取 cookie。
- 不自动点赞收藏评论。
- 不绕过验证码。
- 不确认不发布。

## 第四章：插件使用

主题：把外部能力接进 Codex。

### 4.1 插件是什么

插件可以理解为：

> 一组可安装的能力包，里面可能包含 Skills、工具、MCP、资源和自动化能力。

它解决的问题：

- 让 Codex 使用浏览器。
- 让 Codex 处理文件。
- 让 Codex 连接外部系统。
- 让某类能力变得可复用。

### 4.2 插件和 Skill 的区别

| 项目 | Skill | 插件 |
|---|---|---|
| 核心 | 工作流程 | 能力包 |
| 内容 | Markdown 指令 | Skill + 工具 + MCP + 资源 |
| 适合 | 复用步骤 | 扩展能力 |
| 例子 | 小红书发布流程 | Chrome 控制插件 |

### 4.3 Browser 插件

适合：

- 打开本地网页
- 验证页面布局
- 点击按钮测试交互
- 做截图检查
- 检查移动端/桌面端效果

不适合：

- 使用你的登录态
- 操作私人账号
- 访问需要 Chrome profile 的页面

### 4.4 Chrome 插件

适合：

- 使用你已登录的网站
- 发布后台操作
- 已登录账号的页面检查
- 需要浏览器 cookie/session 的任务

边界：

- 不读取 cookie。
- 不导出 token。
- 不自动完成敏感提交。
- 发布前必须确认。

### 4.5 MCP 和外部数据

可讲：

- MCP 适合连接结构化工具和数据源。
- 比浏览器更稳定。
- 能用 API/MCP 时，不要优先用浏览器硬点页面。
- 浏览器适合 UI 验证，MCP 适合数据和动作接口。

### 4.6 工具选择原则

```text
本地代码修改 → 文件系统 + 命令
本地网页验证 → Browser
已登录网页 → Chrome
结构化外部数据 → MCP/API
重复流程 → Skill
长期项目规则 → AGENTS.md
```

### 4.7 实战案例：Chrome 发布小红书

流程：

1. 检查 Chrome Extension 是否安装。
2. 打开小红书创作后台。
3. 上传封面。
4. 填写标题和正文。
5. 预览内容。
6. 用户确认。
7. 点击发布。

可能问题：

- 扩展没装
- profile 不一致
- 文件上传权限未开
- 页面需要验证码
- 平台自定义组件不响应普通点击

## 第五章：工作流设计

主题：让 Codex 从工具变成系统。

### 5.1 工作流是什么

一句话解释：

> 工作流是把输入、处理、验证、交付固定下来，让每次任务都能稳定复现。

### 5.2 单次任务工作流

```text
理解目标
  ↓
读取上下文
  ↓
制定方案
  ↓
执行修改
  ↓
运行验证
  ↓
总结交付
```

适合：

- 修 bug
- 改页面
- 写脚本
- 整理文档

### 5.3 内容生产工作流

```text
选题
  ↓
目标用户
  ↓
标题
  ↓
正文
  ↓
封面
  ↓
发布
  ↓
复盘
```

可以做成：

- 小红书运营工作流
- 公众号选题工作流
- 短视频脚本工作流
- 知识产品拆解工作流

### 5.4 开发工作流

```text
需求澄清
  ↓
代码阅读
  ↓
最小实现
  ↓
测试验证
  ↓
代码审查
  ↓
交付说明
```

关键原则：

- 先读再改
- 小步提交
- 每步可验证
- 不做无关重构

### 5.5 风险分层

低风险可自动执行：

- 读取文件
- 修改草稿
- 本地语法检查
- 生成文档

中风险要说明：

- 大范围改动
- 新增依赖
- 改配置
- 访问外部网站

高风险必须确认：

- 删除文件
- 发布内容
- 提交表单
- 推送代码
- 支付或购买
- 改线上权限

### 5.6 验证设计

常见验证方式：

- 语法检查
- 单元测试
- 构建
- 浏览器预览
- 页面截图
- 人工确认
- 日志检查

验证提示词：

```text
请在完成后运行最轻量的验证命令。
如果无法验证，请说明原因和残余风险。
```

### 5.7 从提示词到系统

沉淀路径：

```text
一次性提示词
  ↓
常用提示词
  ↓
AGENTS.md
  ↓
Skill
  ↓
插件
  ↓
自动化
```

判断方法：

- 只用一次 → 提示词
- 项目长期规则 → AGENTS.md
- 重复任务流程 → Skill
- 需要工具能力 → 插件
- 需要定时执行 → 自动化

### 5.8 实战案例

案例：维护“小红书内容发布系统”。

Codex 可以负责：

- 生成文章
- 更新工作台
- 生成封面图
- 打开发布后台
- 填写草稿
- 等待最终确认
- 记录复盘字段

用户负责：

- 判断内容方向
- 确认事实准确
- 处理平台验证
- 决定是否发布

## 第六章：多 Agents 部署与协同

主题：复杂任务不要只靠一个脑子。

### 6.1 多 Agents 是什么

一句话解释：

> 多 Agents 是把一个复杂任务拆给多个专门角色，再由主控整合结果。

不是为了炫技，而是为了解决：

- 任务太复杂
- 信息太多
- 需要独立审查
- 需要不同视角
- 需要并行推进

### 6.2 适用场景

适合：

- 竞品分析
- 内容选题规划
- 代码审查
- 产品方案设计
- 课程大纲设计
- 大型文档整理
- 多平台发布计划

不适合：

- 很小的单步任务
- 目标不清的任务
- 没有最终决策人的任务
- 需要高度实时交互的任务

### 6.3 常见角色设计

主控 Agent：

- 拆任务
- 分配角色
- 合并结果
- 做最终判断

研究 Agent：

- 找资料
- 整理事实
- 输出参考依据

执行 Agent：

- 写代码
- 写文案
- 做表格
- 产出初稿

审核 Agent：

- 找漏洞
- 查风险
- 提出改进点
- 防止主观偏差

编辑 Agent：

- 优化表达
- 调整结构
- 提升可读性

### 6.4 协同方式

推荐流程：

```text
主控定义目标
  ↓
拆分角色
  ↓
各 Agent 独立输出
  ↓
主控合并
  ↓
审核 Agent 找问题
  ↓
主控定稿
```

### 6.5 上下文隔离

为什么要隔离：

- 防止互相污染判断。
- 避免所有 agent 都重复一个错误假设。
- 让审核更独立。

做法：

- 研究 Agent 只看资料任务。
- 执行 Agent 只看交付要求。
- 审核 Agent 只看结果和标准。
- 主控 Agent 负责整体信息。

### 6.6 质量控制

审核 Agent 的任务不应该是：

```text
帮我看看这个写得好不好。
```

而应该是：

```text
请从事实错误、逻辑漏洞、合规风险、用户不理解的地方四个角度审查这份内容，并按严重程度排序。
```

### 6.7 常见错误

- 为了多 agent 而多 agent。
- 每个 agent 任务都太泛。
- 没有最终负责人。
- 输出格式不统一。
- 主控没有复核。
- 审核 agent 只会夸，不会找问题。

### 6.8 实战案例：小红书内容生产多 Agent

角色：

1. 选题 Agent：找用户痛点和选题角度。
2. 写作 Agent：生成小红书正文。
3. 标题 Agent：生成 10 个标题并排序。
4. 合规 Agent：检查夸大、虚假、平台风险。
5. 主控 Agent：选择最终版本并发布前确认。

流程：

```text
用户给方向
  ↓
选题 Agent 拆角度
  ↓
写作 Agent 写正文
  ↓
标题 Agent 做标题
  ↓
合规 Agent 审查
  ↓
主控 Agent 定稿
  ↓
用户确认发布
```
